# EduAgent 多智能体系统边界决策

## 1. 核心目标

把新版 ConversationAgent 的自然对话能力，和旧版 workflow 的稳定执行链路结合起来。

最终希望系统做到：会聊天、能调度、能生成、不会夺舍、不会擅自执行、不会假装执行成功。

## 2. 主 Agent / ConversationAgent 的定位边界

主 Agent / ConversationAgent 是最高优先级调度中心，负责：

- 理解用户意图。
- 判断当前条件是否足够。
- 决定是否调用子 Agent。
- 决定调用哪些子 Agent。
- 判断是否需要先问用户确认。
- 统一生成最终回复。

子 Agent 不应该自己抢流程，也不应该直接决定最终用户话术。

主 Agent 可以拦截后续 Agent，但必须有约束。

可以拦截的情况：

- 用户只是闲聊。
- 用户只是补充画像。
- 用户只是表达学习意向。
- 用户请求不明确。
- 生成条件明显不足。

不能无声拦截的情况：

- 用户明确要求生成学习方案。
- 用户刚刚确认“可以 / 就这样 / 按这个来”。
- 当前 `readyToPlan=true` 且用户确认生成。
- 后端接口或测试明确要求 `full_workflow`。

拦截时必须告诉用户当前为什么不执行、还缺什么、下一步可以怎么做。

ConversationAgent 循环执行时，每一步必须产生新的有效状态；若连续两次没有新增状态，或同一 Agent 重复失败，应停止循环并向用户说明当前卡点，避免无限或无效循环。

## 3. 学习方案生成边界

学习方案生成规则：

- 不自动生成学习方案。
- 只有用户明确要求或确认后才生成。
- 信息足够时可以提示“已经可以生成初版”，但必须等用户确认。

示例：

- “我想学习数据结构” => 只记录目标，不生成。
- “我是大三，我学过 C 语言” => 更新画像，不生成。
- “开始生成学习方案” => 生成。
- “按这些信息生成” => 生成。
- 系统问“要不要生成”，用户答“可以” => 生成。

生成学习方案只有两个硬条件：

- 学习对象明确，例如数据结构、微积分、Python、链表、考研英语。
- 用户明确要求或确认生成。

以下信息不是硬条件，有就用，没有就默认假设，并在方案里说明：

- 已有基础。
- 学习目标。
- 学习周期。
- 每日学习时间。
- 学习偏好。
- 薄弱点。
- 资源偏好。
- 考试 / 项目 / 面试目标。

不能再用“画像完整度 7/7”硬卡生成。

用户明确要求生成但条件不足时：

- 如果连学习对象都不知道，必须先问。
- 如果只缺时间、偏好、薄弱点等非关键项，可以询问，也可以让用户确认按默认假设生成初版。
- 不能直接拒绝。
- 不能假装生成。
- 不能擅自按默认生成，仍然需要用户确认。

## 4. 确认词边界

“可以 / 就这样 / 按这个来 / 好”只有在有明确上下文时才算确认生成。

例如上一轮系统问：“这些信息已经可以生成初版，要开始吗？”用户回答“可以”，算确认生成。

如果用户突然只说“可以”，没有上下文，系统应追问：“你是想让我开始生成学习方案，还是继续补充信息？”

## 5. 子 Agent 调用边界

主 Agent 可以单独调用某个子 Agent，但要先检查前置条件。

示例：

- 用户要诊断薄弱点 => 可以调用 DiagnosisAgent，但要知道诊断哪个课程或知识点。
- 用户要推荐资源 => 可以调用 ResourceAgent，但要知道课程 / 阶段 / 知识点。
- 用户要学习计划 => 调用 Planner 或 `full_workflow`。
- 用户只是闲聊或补充信息 => 不调用子 Agent。

不能盲目调用子 Agent。条件不够时，主 Agent 先追问最少必要信息。

子 Agent 职责边界：

- ProfileAgent：只负责抽取和更新用户画像。
- DiagnosisAgent：只负责诊断薄弱点，并给 `evidence_chain`。
- PlannerAgent：只负责生成结构化学习路径。
- ResourceAgent：负责推荐和生成学习资源；在具备学习路径或阶段信息等前提条件时，需要将资源绑定到对应阶段并标注来源；若用户强行要求直接推荐资源，则可以不绑定阶段，但仍需说明来源。
- ReviewAgent：只负责复习和掌握度回顾。

子 Agent 不负责最终回复，不决定是否生成，不抢用户可见话术。

## 6. 最终回复权边界

最终用户回复由 ConversationAgent / MainAgent 负责，但前提是它必须读取真实 pipeline 执行结果。

硬规则：

- Planner 没跑，就不能说学习路径已生成。
- Resource 没跑，就不能说资源已推荐。
- Diagnosis 没跑，就不能说已经完成诊断。
- 不能凭感觉说“我已经生成了”。

推荐结构：

1. ConversationAgent 第一阶段：判断意图、决定 action。
2. ConversationAgent 内部增加循环判断机制：每次执行完一步后重新判断当前状态，是直接回复用户，还是继续调用其他 Agent；该循环最多执行 5 次，防止无限循环。
3. Orchestrator：根据 ConversationAgent 的决策真实调用子 Agent。
4. 子 Agent：返回结构化结果。
5. ConversationAgent 最终阶段：在循环结束后，根据真实执行结果统一生成用户回复。

防止“夺舍”的规则：

- 用户可见回复只能有一个出口。
- 旧模板、fallback、`product.py`、子 Agent 都不能直接抢最终话术。
- 所有回复都要保持 MainAgent / ConversationAgent 的统一风格。
- 每轮内部记录 `reply_source`、`action`、`pipeline_executed`、`agents_run`、`final_reply_owner`。

## 7. fallback / 旧人格边界

用户可见层不要旧人格。

旧系统话术不应该再出现：

- “画像完整度 2/7”。
- “请补充以下 7 项信息”。
- “当前画像信息如下”。
- 生硬表格式追问。

fallback 要保留，但只能作为后台技术兜底，且仅负责输出调试性或技术类信息，完全不面向用户。

fallback 可以做：

- 结构化识别 action。
- 判断是否是明确生成请求。
- 判断是否是确认词。
- 返回 `missing_info`。
- 返回 `llm_unavailable` / `schema_error` 等状态。
- 输出调试日志、错误信息、执行路径等开发者可读信息。

fallback 不能做：

- 直接对用户说话。
- 输出旧模板。
- 抢最终回复。
- 把明确生成请求误判成 `action=none` 后静默截断。
- 假装执行成功。
- 生成任何面向用户的自然语言回复。

fallback 只允许输出后台结构化状态和开发者调试日志，不允许生成任何用户可见自然语言。

MainAgent / LLM 出错时按分级恢复：

1. MainAgent 调用失败后，系统内部自动重试，最多 5 次。
2. 如果返回格式错误，内部要求模型按 schema 修复，这不是对用户说。
3. 如果仍然失败，提示用户可以重试 / 刷新 / 选择备用模型。
4. 不应自动切回旧人格。
5. 备用模型也必须使用同一套 MainAgent prompt 和同一套回复风格。
6. fallback 是最后的后台兜底，不直接面向用户。

## 8. 多轮画像合并边界

已有画像信息只能保守覆盖。

例如用户已经说“我想学微积分”，后面又说“主要是复习”，不能把 `target_course` 从“微积分”覆盖成“复习”。

正确合并：

- `target_course = 微积分`
- `learning_goal = 复习 / 期末复习`

只有用户明确说“不是学微积分了，改成数据结构”，才允许覆盖 `target_course`。

建议保护字段：

- `target_course`
- `user_background`
- `prior_knowledge`
- `learning_goal`
- `learning_period`
- `daily_time`
- `weak_points`
- `preference`

合并原则：

- 明确信息优先。
- 新信息如果只是补充，不覆盖旧字段。
- 新信息和旧信息冲突时，主 Agent 先确认。
- 不把“复习 / 高分 / 入门 / 考试”误当课程名。

## 9. Planner 边界

PlannerAgent 不应该只输出“补齐基础 / 攻克核心 / 快速复盘”，必须按课程内容阶段生成。

数据结构示例：

- 复杂度、数组、链表。
- 栈、队列、递归。
- 树、二叉树、遍历。
- 图、查找、排序。
- 综合练习、错题复盘、应用。

微积分示例：

- 函数、极限、连续。
- 导数、微分。
- 导数应用。
- 积分基础。
- 综合题型、期末复盘。

Planner 缺信息时：

- 不能因为缺诊断数据就输出“无诊断数据”。
- 不能把 `fallback_rule`、`unknown` 等内部字段展示给用户。
- 可以使用默认假设。
- 必须把假设写清楚。
- 后续用户补充信息后，可以调整计划。

时间解析边界：

- Planner 必须正确处理“两天 / 2天 / 二天 / 一周 / 两周 / 一个月 / 30天 / 每天三小时 / 周末休息”。
- 时间很紧时，`priority_basis` 应包含 `time_budget`。
- 时间很紧时，`risk_flags` 应包含 `time_budget_tight`。
- 学习计划应更偏重点突破，而不是完整铺开。

## 10. Diagnosis 边界

DiagnosisAgent 可以诊断的情况：

- 用户明确说自己哪里不会。
- 用户上传错题 / 答题结果。
- 用户上传自己的作答结果时，可以触发判卷能力，对答案进行评估、打分并给出解析。

用户没有提供任何证据时，不能瞎编薄弱点。

诊断结果必须包含：

- `weak_knowledge_points`
- `evidence_chain`
- `confidence`
- `needs_more_evidence`
- `recommended_next_actions`
- `risk_flags`

诊断必须说明依据：

- 用户自述。
- 测验结果。
- 错题。
- 学习记录。
- 资源完成情况。

DiagnosisAgent 对用户作答进行判卷时，必须标注依据；对开放性题目或证据不足的情况，只能给参考性评价，不能给绝对化结论。

## 11. Resource 边界

资源推荐的前置条件应由主 Agent 统一判断，而不是由 ResourceAgent 在内部自行拒绝请求。

最低必要条件：必须明确课程或知识点。

其他信息，如用户目标、学习阶段、学习路径中的 stage，属于增强信息，有则更好，但不作为硬性阻断条件。

如果当前信息不足以进行合理推荐，应由主 Agent 先进行追问或补充信息，而不是让 ResourceAgent 直接拒绝执行。

ResourceAgent 在没有学习路径或阶段信息时，可以进行通用资源推荐，但必须标记为通用推荐，不能伪装成基于个人学习路径的阶段化推荐。

ResourceAgent 输出必须包含：

- resource title
- type
- `related_stage_id`
- `knowledge_points`
- reason
- source / provenance
- `quality_status`

不能伪造来源。没有 RAG 证据时，要标记 fallback 或 generated，不要装作来自真实资料。

## 12. Orchestrator 边界

MainAgent 是用户意图和对话总控，Orchestrator 是执行调度器。

Orchestrator 必须返回真实执行状态给 MainAgent，但不直接决定最终话术。

Orchestrator 不能做：

- 不能因为 `action=none` 就静默截断明确生成请求。
- 不能让测试 / `full_workflow` 调用被 ConversationAgent 误杀。
- 不能在 pipeline 没跑时返回“已生成”。
- 不能吞掉 Agent 错误不告诉 MainAgent。

## 13. Debug / 测试边界

开发环境至少保留 debug 字段。

每轮应记录：

- `action`
- `confidence`
- `should_run_pipeline`
- `skip_pipeline`
- `skip_reason`
- `agents_run`
- `pipeline_executed`
- `learning_path_created`
- `resources_created`
- `final_reply_owner`
- `reply_source`
- `fallback_used`
- `llm_retry_count`

固定 smoke 测试：

- 场景 A：只表达学习意向，不自动生成。输入包括“我是大三的 / 我想学习数据结构 / 我学过 C 语言”。预期：记录画像，不生成方案。
- 场景 B：明确要求生成。已有大三、数据结构、C 语言，用户说“开始生成学习方案”。预期：生成学习路径。
- 场景 C：完整微积分画像。用户说“我想一个月学微积分 / 高中导数学过，极限没学 / 每天三小时，周末休息 / 我是大一软件工程 / 目标期末高分 / 开始生成学习方案”。预期：生成极限、导数、积分、综合训练阶段。
- 场景 D：确认词。系统问是否生成，用户说“可以”。预期：生成。
- 场景 E：无上下文确认词。用户直接说“可以”。预期：追问，不生成。

## 14. 前端展示边界

前端应该展示：

- 统一 `final_reply`。
- 学习路径、资源、诊断卡片。

前端不要直接展示内部脏字段，例如 `fallback_rule`、`unknown`、`无诊断数据`、`source=conversation_only`。

debug 展示规则：

- 普通用户不显示。
- 开发模式可以显示。
- 比赛演示时可以隐藏。
- 调试时用于判断是否真的跑了 Agent。

## 15. 模型 Provider 边界

允许备用模型，但必须由用户确认或系统配置控制。

规则：

- 主模型失败后先内部重试。
- 仍失败时询问用户是否使用备用模型。
- 备用模型必须使用同一套 MainAgent prompt。
- 不允许切换到旧模板人格。
- 不同模型输出必须经过同一 schema 校验。

## 16. 最终总原则

系统对用户只能表现为一个统一的 MainAgent。

MainAgent 负责：

- 理解。
- 判断。
- 询问。
- 调度。
- 读取真实执行结果。
- 统一回复。

子 Agent 负责：

- 执行具体任务。
- 返回结构化结果。
- 不直接抢最终回复。

fallback 负责：

- 后台兜底。
- 不直接说话。
- 不恢复旧人格。

学习方案生成原则：

- 不自动生成。
- 学习对象明确 + 用户确认生成，就生成初版。
- 其他画像信息是优化项，不是硬阻断。
- 缺失信息用默认假设，但要说明。
- 条件不足时先问，不假装执行。

最终目标：新版负责聪明，旧版负责稳定，但用户永远只看到一个统一、自然、可信的智能学习助手。
